# 《Clean Code》关键内容提炼

这本书最核心的主张不是“把代码写漂亮”，而是：把代码写成后来者能快速读懂、敢于修改、持续清理的样子。

## 关键命题

- 可读性优先于作者的书写便利；代码主要是给未来的维护者读的。
- 坏代码最贵的成本不是“丑”，而是让修改越来越慢、越来越危险。
- 重构不是额外活动，而是日常开发的一部分；看到坏味道就应顺手清理。

## 关键内容

### 1. 命名不是修饰，而是主要表达手段

- 名字要直接传达意图，减少读者推断成本。
- 避免无信息量词汇、晦涩缩写、类型噪音和误导性命名。
- 如果上下文已经提供语义，不要在局部名字里重复父级语义。
- 好命名应让调用点接近可读句子，而不是迫使读者翻实现猜意思。

### 2. 函数要保持单一意图和统一抽象层次

- 一个函数应围绕单一意图展开，不要把业务编排、状态更新、校验和底层细节混在一起。
- 参数越少越好；布尔参数、输出参数和隐藏时序约束都是风险信号。
- 副作用必须显式，不能让名字像查询、行为却像修改。
- 书里强烈偏好小函数，目的是把控制流写得更像叙述，而不是鼓励机械切碎。

### 3. 注释不是默认解释器

- 最优先的做法是让命名、结构和抽象本身表达意图。
- 注释主要用于代码难以自然表达的信息：设计意图、契约、警告、边界条件、历史包袱。
- 废话注释、过期注释和与代码重复的注释会制造额外噪音。
- 这本书整体上对注释比较克制，倾向于先改代码，再决定是否需要注释。

### 4. 错误处理必须和主流程分离

- 错误路径不能把主逻辑淹没。
- 倾向于使用异常或更明确的错误语义，而不是到处传播模糊错误码。
- 错误信息要可诊断，但不能把业务代码写成一层层防御性分支。
- 边界检查、空值处理和异常语义属于设计，不是“最后补上”的细节。

### 5. 类和模块应按变化原因组织

- 类要高内聚，不要把彼此独立的变化理由塞在一个结构里。
- 区分“有行为的对象”和“仅承载数据的结构”；接口和实现不要彼此污染。
- 重复逻辑应收敛为单一事实来源，避免知识散落导致修改时多处同步。
- 模块组织的目标是减少读者需要同时理解的东西。

### 6. 测试是重构安全网

- 测试的价值首先在于支撑持续清理，而不是只做验收。
- 书里强调 F.I.R.S.T. 测试特性：快、独立、可重复、自校验、及时。
- TDD 被放在整洁代码实践内部，因为它能逼迫设计在可验证的小步里演化。
- 没有可靠测试，很多“想重构”最后都会退化为“不敢动”。

### 7. 持续清理比一次性重写更现实

- Boy Scout Rule：离开时让代码比来时更干净。
- 坏味道一旦积累，就会拖慢后续所有开发；越晚处理，代价越大。
- 真正的整洁不是一次整理出来，而是在每次修改里不断还债。

## 这本书最值得带走的判断句

- 这个名字是否让调用点不用读实现也能大致理解意图？
- 这个函数是否同时混合了多个抽象层次或副作用？
- 这个模块里的代码是否因为多个变化原因纠缠在一起？
- 这段注释是在补充代码难以表达的信息，还是在替代码翻译？
- 如果今天要改需求，测试是否足以支撑我安全重构？

## 容易被机械化误读的部分

- “函数越短越好”不应被执行成无止境拆分。
- “少写注释”不等于“不写契约和意图”。
- “单一职责”不是把每一行都抽成独立函数，而是按变化原因切边界。
