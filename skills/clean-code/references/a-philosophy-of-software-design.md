# 《软件设计哲学》关键内容提炼

这本书的中心不是代码风格，而是一个更上游的问题：怎样让系统更容易理解和修改。Ousterhout 把设计优劣几乎都收束到“复杂度是否下降”这个问题上。

## 关键命题

- 好设计的首要目标是降低复杂度，而不是增加结构名词。
- 复杂度不是“代码多”，而是后来者必须同时知道的东西太多。
- 设计应当优先减少读者的认知负担，而不是只让局部代码看起来整齐。

## 复杂度的三个直接表现

- Change amplification：改一个需求却要改很多地方。
- Cognitive load：为了改一处代码，脑中必须同时装下太多前置知识。
- Unknown unknowns：你甚至不知道还需要去哪里找关键事实。

## 复杂度的主要来源

- 依赖过多：一处变化会牵动很多模块。
- 信息泄漏：本应封装在模块内部的知识被迫由调用方理解。
- 不一致：相似概念在不同位置用不同规则、不同命名、不同错误语义表达。

## 关键内容

### 1. Deep Module 是核心结构判据

- 好模块应该用小接口隐藏大量实现复杂度。
- 如果一个模块拆出来后只是多了一层名字和转发，它就是浅模块。
- 判断拆分是否成功，不看文件数，而看调用方是否真的少知道了东西。

### 2. 复杂度要向下压，而不是向上传

- 调用方应依赖稳定接口，而不是理解内部流程、时序或目录结构。
- Pass-through methods、镜像 API、只转发不抽象的层，通常是在把复杂度继续往外扩散。
- 真正好的抽象会让使用者只记住接口，不必记住内部细节。

### 3. 通用机制通常比特化机制更深

- 过度针对当前场景的模块往往接口很窄、寿命也很短。
- 如果一个机制能覆盖更广的问题，同时接口仍然简单，它通常比临时拼装的特化方案更深。
- 第二版特别加强了这一点，把“General-Purpose Modules are Deeper”作为重点展开。

### 4. 不要过度拆分相关逻辑

- 拆分只有在接口比原代码更简单时才成立。
- 如果两个函数必须来回跳着读、一起记忆、一起推理，它们就是 entangled，拆开反而更难懂。
- 因此“更多函数”不自动等于“更好设计”；很多时候相关逻辑放在一起更清楚。

### 5. 注释是设计工具，不只是补丁

- 接口注释负责定义契约，让调用方知道可以依赖什么。
- 实现注释负责写出代码本身不明显的关键事实、设计意图和反直觉约束。
- 数据结构注释负责说明字段含义、生命周期和不变量。
- 书里的立场不是“注释越多越好”，而是“缺少关键注释会直接增加复杂度”。

### 6. 通过消灭特殊情况来降复杂度

- 特殊分支、例外协议和零碎补丁会迅速放大认知成本。
- 一个重要策略是“define errors out of existence”：通过重新设计接口、数据模型或流程，把原本需要特殊处理的情况变成常规路径。
- 好设计经常表现为：调用方根本不需要知道某类异常细节曾经存在。

### 7. 设计需要战略性思考，而不是纯战术推进

- Tactical programming 关注“先把眼前改动做完”；Strategic programming 会花更多时间思考长期复杂度。
- 设计不是等代码写完再收拾，而是在写之前就考虑边界、接口、依赖和信息隐藏。
- 第二版新增 “Decide What Matters”，强调必须先识别真正影响复杂度的因素，再集中火力处理它们。

### 8. 一致性本身就是降复杂度的手段

- 一致的命名、错误语义、接口风格和结构约定，会降低读者切换上下文的成本。
- 不一致会制造额外判断：同样的事情，这里是不是又有一套不同规则？

## 这本书最值得带走的判断句

- 这次拆分之后，调用方是真的少知道了东西，还是只是多认识了几个文件？
- 这个接口是否隐藏了复杂度，还是把复杂度改头换面后暴露出来？
- 改一个需求时，为什么要改这么多地方？是不是 change amplification 已经出现？
- 这段代码难读，是因为逻辑本身复杂，还是因为关键信息散落在太多地方？
- 这里需要的是再拆一层，还是把紧密耦合的逻辑收回到一起？

## 与《Clean Code》的公开分歧焦点

- 它更警惕“小函数”被机械执行成过度拆分。
- 它更强调接口注释和实现注释在降低复杂度中的作用。
- 它更偏向先做设计判断，再推进实现，而不是把开发过程过度战术化。
